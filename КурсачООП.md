**Сетевой протокол:** TCP (Host-Authoritative).

## 1. Описание

**Pixel Dungeon: University Edition** — это рогалик (roguelike), в котором игроки в роли студентов сражаются с воплощениями учебных страхов (Дедлайн, Тревожность, Сравнение). Игра поддерживает одиночный режим с сюжетом и кооператив до 4 человек. Особенностью является процедурная генерация уровней и динамическая сложность, зависящая от параметра «Послушания Программе».

## 4. Игровой цикл

Игровой цикл определяет порядок обновления состояния игры в каждом кадре.
В проекте используется событийная модель Qt, где основной цикл выполняется внутри QApplication.

Основная логика симуляции реализована в классе World, а управление состояниями приложения выполняется классом Game.

### 4.1 Инициализация приложения

Точка входа программы находится в main.cpp.
```
int main(int argc, char* argv[])
{
    QApplication app(argc, argv);

    Game game;
    game.show();

    return app.exec();
}
```
После запуска происходит:

- создание объекта QApplication

- создание основного объекта игры Game

- отображение главного окна

- запуск основного цикла Qt (app.exec())

После этого управление передаётся системе событий Qt.

### 4.2 Управление состояниями игры
Класс Game выступает центральным контроллером приложения и управляет состояниями игры.
Основные состояния:

1. Главное меню

2. Лобби

3. Одиночная игра

4. Мультиплеер

5. Игровой процесс

Текущее состояние хранится в переменной:
```
GameState m_currentState;
```
Для переключения экранов используется QStackedWidget.

### 4.3 Обработка ввода игрока

Ввод пользователя обрабатывается в классе GameView.

Игрок может взаимодействовать с игрой через:

- клавиатуру

- мышь

- игровые действия

Результатом обработки ввода становится изменение состояния:

-позиции игрока

-направления движения

-атак

-взаимодействий с объектами.

### 4.4 Обновление игрового мира
Основная симуляция выполняется в классе World.

Главный метод обновления:
```
World::update(float deltaTime,
              Player* localPlayer,
              const QHash<quint32, Player*>& remotePlayers,
              bool isHost)
```

| Параметр        | Описание                          |
| --------------- | --------------------------------- |
| `deltaTime`     | время между кадрами               |
| `localPlayer`   | локальный игрок                   |
| `remotePlayers` | список удалённых игроков          |
| `isHost`        | является ли текущий клиент хостом |

Метод вызывается каждый кадр игрового цикла.

### 4.5 Обновление объектов мира

Во время обновления мира происходит обновление всех игровых объектов.

#### Предметы
```
for (auto& item : m_droppedItems)
    item->update(deltaTime);
```
#### NPC
```
for (auto& upperclassman : m_upperclassmen)
    upperclassman->update(deltaTime);
```
#### Манекен
```
for (auto* dummy : m_dummies)
    dummy->update(deltaTime);
```
Также обновляются двери и другие интерактивные элементы уровня.

### 4.6 Проверка столкновений и боевой системы
Во время обновления мира происходит проверка столкновений и нанесения урона.

Например, при атаке оружием:
```
for (auto& swing : m_weaponSwings)
    for (auto* dummy : m_dummies)
        if (swing->checkDummyHit(dummy))
            dummy->takeDamage(...);
```
Если столкновение обнаружено:

- применяется урон

- обновляется состояние объекта

- могут запускаться визуальные эффекты.

### 4.7 Сетевая синхронизация

В мультиплеере используется класс NetworkManager.

Применяется модель:

Host → authoritative server
Clients → получают обновления состояния

### 4.8 Итоговый цикл кадра

Каждый игровой кадр выполняется в следующем порядке:

1. Обработка пользовательского ввода
2. Обновление состояния игрока
3. Вызов World::update()
4. Обновление NPC и объектов
5. Проверка столкновений
6. Применение урона
7. Сетевая синхронизация
8. Отрисовка сцены

```
Game
↓
GameView
↓
World
↓
Entities
↓
NetworkManager
```

## 6. Алгоритмы и их обоснование

### 6.1. Процедурная генерация подземелий (DungeonGenerator.cpp)
**Используемый алгоритм:** Grid-based Randomized Growth (на базе BFS-подобного распространения).
- **Суть:** Выбирается центральная точка, затем случайным образом пристраиваются соседние комнаты, проверяя отсутствие перекрытий. После строится путь к самой дальней «тупиковой» комнате (End Room).
- **Почему это лучше:** В отличие от простого случайного блуждания (Random Walk), этот метод гарантирует связность (игрок всегда может дойти до босса) и позволяет легко контролировать плотность комнат и длину коридоров.

### 6.2. Обнаружение коллизий с вращающимися объектами (Door.cpp)
**Используемый алгоритм:** SAT (Separating Axis Theorem) — Теорема о разделяющей оси.
- **Суть:** Для проверки столкновения игрока с дверью, которая анимируется (поворачивается на угол до 90°), используется проекция вершин выпуклых многоугольников на их нормали.
- **Сравнение:** Обычные AABB (Axis-Aligned Bounding Boxes) дают огромную погрешность при вращении. SAT позволяет игроку «тереться» о дверь во время её открытия без застревания в невидимых хитбоксах.

### 6.3. Сетевая синхронизация (NetworkManager.cpp / WorldState.cpp)
**Используемый алгоритм:** Snapshot Interpolation (Интерполяция состояний).
- **Суть:** Хост отправляет полные или дельта-состояния мира каждые 33мс. Клиент не перемещает объекты мгновенно, а плавно интерполирует (Lerp) их позиции между двумя последними пакетами.
- **Почему это лучше:** Это компенсирует джиттер (дрожание) TCP-пакетов. Даже при задержках в сети движение врагов на экране клиента остается плавным, а не «телепортирующимся».

### 6.4. Логика ИИ врага «Сравнение» (ComparisonEnemy.cpp)
**Используемый алгоритм:** Dynamic Archetype Mirroring.
- **Суть:** Враг анализирует инвентарь и параметры игрока в радиусе агро, копирует его статы и оружие, а затем выбирает стиль поведения (отступление для дальнего боя или агрессивный натиск для ближнего).
- **Сравнение:** Это на порядок сложнее обычного «преследуй и бей». ИИ адаптируется под билд игрока, заставляя того сражаться с собственной копией.

## 7. Сетевой протокол

Пакет состоит из заголовка (5 байт) и полезной нагрузки (Payload).
`[Type (1 byte)] [Size (4 bytes)] [Data (N bytes)]`

**Ключевые типы сообщений (NetMsgType):**
- `FullWorldState (3)`: Полная загрузка уровня для клиента.
- `PlayerUpdate (10)`: Позиция и статы игрока.
- `WeaponSwing (50)`: Передача события атаки (направление, тип оружия).
- `BossDefeated (111)`: Триггер окончания уровня.

## 8. Игровые механики

### 8.1. Система способностей (Statue.cpp)
Игрок может молиться у статуй за золото:
- **Космонавт:** Дарует `SpeedBoost` (рывок ветра).
- **Пётр I:** Дарует `DamageShield` (уменьшение входящего урона на 50%).
- **Гидробашня:** Дарует `AttackSpeed` (удвоение темпа атаки).

### 8.2. Эффект «Тревожности» (AnxietyEnemy.cpp)
При попадании снаряда «Тревожности» на игрока накладывается дебафф, который математически искажает вектор прицеливания (`getActualAimDirection` в `Player.cpp`). Направление выстрела начинает вибрировать по синусоиде, имитируя дрожь рук.

## 9. Установка и запуск

### Сборка (настольная версия)
1. Установите **Qt 6.5+** и **CMake**.
2. Откройте `CMakeLists.txt` (или `.pro` файл) в Qt Creator.
3. Соберите проект (Ctrl+B).

### Запуск мультиплеера
1. **Хост:** Выберите "Multiplayer" -> "Host Game". Сообщите свой IP и порт (по умолчанию 12345) друзьям.
2. **Клиент:** Выберите "Multiplayer" -> "Join Game", введите IP хоста.
3. После подключения всех участников хост нажимает "Start Game", запуская 10-секундный отсчет и вступительную кат-сцену.
